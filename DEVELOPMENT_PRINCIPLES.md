# Fast GUI LLM Orchestration: 개발 원칙 가이드 (이 프로젝트 전용)

이 문서는 본 프로젝트의 핵심 설계 철학이자 개발자가 반드시 준수해야 할 '개발 헌법'입니다. 
Antigravity(AI 코딩 어시스턴트)는 모든 작업 시작 전 이 가이드를 반드시 참조하며, 임의로 설계를 확장하지 않습니다.

---

## 🛡️ 1순위 원칙: "한 걸음씩, 그리고 100%의 이해"
*   **단계별 검수:** 많은 단계를 한꺼번에 뛰어넘지 않는다. 최소 단위의 개발이 끝날 때마다 사용자에게 검수를 받고 다음 단계로 넘어간다.
*   **No Blackbox:** 사용자가 모든 로직을 100% 읽고 이해할 수 있어야 한다. 복잡하거나 마법 같은 자동화 라이브러리(Registry, DI Container 등)를 배제하고, 명시적이고 단순한 코드로만 구현한다.
*   **명시적 코드:** `switch-case`나 `if-else` 같이 눈에 보이는 흐름을 지향하며, 추상화가 과해져서 구조를 파악하기 힘들게 만들지 않는다.

## ⚙️ 2순위 원칙: "과잉 설계 방지 (YAGNI)"
*   **미래 방지:** "나중에 필요할 것 같아서" 미리 기능을 추가하지 않는다. 지금 당장 동작하는 데 꼭 필요한 최소한의 필드와 메서드만 생성한다.
*   **단순성 유지:** 복잡한 디자인 패턴보다 직관적인 로직을 우선한다.

## 🏗️ 3순위 원칙: "물리적/논리적 결합도 분리"
*   **Decoupling:** UI 레이어(React)와 실행 엔진 레이어(Schema/Task)를 철저히 분리한다. 
*   **이동 가능성:** 엔진 레이어는 언젠든 그대로 떼어내어 백엔드 서버(JS/Python)로 옮겨 배포할 수 있도록 UI 의존성을 0%로 유지한다.

## 🔀 4순위 원칙: "데이터 중심 설계"
*   **Schema & Task 분리:** 상태는 Schema가 들고 있고, Task는 상태 없이(Stateless) 작업만 수행하는 Kafka적 아키텍처를 유지한다.
*   **Universal Spec:** 데이터 구조(JSON)는 범용적으로 설계하여 추후 다른 언어(C, Java 등)로 재구현(Reshape)하기 용이하도록 만든다.

---
## 5순위 원칙: "작업 범위 및 코드 무결성 유지"
*   **범위 이탈 금지:** 오직 지시받은 기능 구현이나 수정 범위 내에서만 작업한다.
*   **코드/주석 보존:** 지시 내용과 무관한 기존의 주석 삭제, 코드 라인 변경, 서식 수정(Formatting) 등을 절대 임의로 수행하지 않는다. 
*   **최소 침습 수정:** 코드의 동작에 꼭 필요한 최소한의 라인만 수정하여 사용자가 변경 사항을 추적하기 용이하게 한다.
