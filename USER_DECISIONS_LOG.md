# User Decision Log: Fast GUI LLM Orchestration
본 프로젝트의 핵심 기획 및 기술적 의사결정 기록물입니다.

## 프로젝트 계기 (Background)
> "2026년 봄학기, AI와 사회에 대한 산업디자인학과 강의를 듣던 도중, AI 기술을 사람들이 얼마나 올바르게 접하고, 사용하는 경험을 쌓아야지 그에 대한 이해도가 높아질 수 있는지 배우게 되었습니다. 저 또한 과거 딥러닝과 머신러닝이 인기이던 동안, Google의 Teachable Machine을 통해 다가가기 어렵기만하던 기술을 접하고, 사용하고, 그 잠재력을 실감할 수 있었습니다.
이제 AI를 통한 LLM과 에이전트의 시대가 온 지금, 비전공자들과 비전문가들 또한 LLM을 기반으로 한 에이전트와 오케스트레이션의 쉽고 간단한 접근을 통해 빠른 프로토타입을 만들고 이용하는 경험이 필요하다고 느꼈고, GUI를 기반으로 노드 단위의 LLM 모델과 로직 노드들을 조작하여 자신만의 필요에 맞는 LLM Agent를 만들게 되길 바랍니다."

---

## 📅 2026-04-09
- **타겟 유저 설정:** 비개발자와 프롬프트 엔지니어를 제1 타겟으로 설정.
- **브랜딩 컨셉:** 구글 'Teachable Machine'과 같은 극강의 직관성을 가진 GUI 지향.
- **핵심 킬러 기능:** 생성된 결과물을 플랫폼 종속 없이 단독 실행 가능한 **순수 파이썬 파일(.py)로 Export** 하는 기능 확정.
- **프로젝트 투명성:** 모든 진행 상황과 철학을 `README.md` 및 본 의사결정 로그에 가감 없이 기록하여 오픈소스적 가치 확립.

## 📅 2026-04-10
- **서비스 제공 형태:** 서버 비용 부담을 없애고 오픈소스 철학을 지키기 위해, 사용자 본인의 API Key를 사용하는 **'100% 클라이언트 기반 서버리스 웹앱'** 형태로 개발 방향 결정.
- **철저한 레이어 분리:** UI UI 컴포넌트 로직과 핵심 실행 엔진(Schema/Task)을 논리적으로 완전히 분리하여 설계 (언젠든 백엔드로 도려내어 재배포가 가능하도록 함).
- **이벤트 기반(Reactive) 아키텍처:** 폴링 방식이 아닌, 상태를 가진 `Schema`와 상태 없이 비동기로 작업을 수행하고 사라지는 `Task Consumer` 그룹으로 이원화.
- **설계 배경:** 전산망개론(KENS) 프로젝트의 책임 분리, 게임 개발의 Controller-View 페어 구조, Kafka의 Topic-Consumer 모델에서 인사이트를 얻음.

## 📅 2026-04-11
- **기술 스택 확정:** 프론트엔드는 `JS + React + React Flow`를 사용하며, 엔진은 100% 이해 가능한 명시적인 코드로만 직접 구현하기로 결정.
- **다국어 이식성(Portability) 보장:** 특정 라이브러리(Blackbox) 대신, Java, Python, C 등 어떤 언어로도 핵심 실행 구조를 **재구성(Reshape)** 할 수 있도록 보편적인 데이터 규격(JSON) 지향.
- **Task 관리 방식:** 거창한 스케줄러 대신, JS의 비동기(Async) 런타임을 활용하고 Topic 수신 시 즉석에서 인스턴스를 생성/파괴하는 동적 매니저 방식 채택.

## 📅 2026-04-16
- **장기적 확장 비전:** 향후 VLA(Vision-Language-Action) 및 복잡한 비디오 파이프라인 노드를 수용할 수 있도록 노드 데이터 구조의 유연성 확보 결정.
- **대용량 콘텐츠 처리 전략(Asset Store 방식):** 영상/이미지 같은 무거운 데이터는 프론트엔드 메모리(Asset Store)에 중앙 집중 관리하며, 노드 간 데이터 전달 시에는 실제 데이터가 아닌 **'가벼운 자산 ID(Asset ID/Reference)'**만 Topic으로 주고받아 엔진의 부하를 최소화하고 UX를 극대화하기로 결정.
- **실행기 설계 보완:** LLMNode 같이 설정값이 복잡한 노드를 위해 `NodeDataPayload`에 노드 고유 세팅값(`config`) 필드 추가 확정.
