# User Decision Log: Fast GUI LLM Orchestration
주요 기획 및 기술 의사결정 기록

## 프로젝트 계기 (Background)
> "2026년 봄학기, AI와 사회에 대한 산업디자인학과 강의를 듣던 도중, AI 기술을 사람들이 얼마나 올바르게 접하고, 사용하는 경험을 쌓아야지 그에 대한 이해도가 높아질 수 있는지 배우게 되었습니다. 저 또한 과거 딥러닝과 머신러닝이 인기이던 동안, Google의 Teachable Machine을 통해 다가가기 어렵기만하던 기술을 접하고, 사용하고, 그 잠재력을 실감할 수 있었습니다.
이제 AI를 통한 LLM과 에이전트의 시대가 온 지금, 비개발자들과 비전공자, 비전문가들 또한 LLM을 기반으로한 에이전트와 오케스트레이션의 쉽고 간단한 접근을 통해 빠른 프로토타입을 만들고 이용하는 경험이 필요하다고 느꼈고, 문득 떠오른 아이디어와 함께 GUI를 기반으로 노드 단위의 LLM 모델과 로직 노드, 문자열 노드, 포맷 노드 등을 조작하여 쉽고 빠르게 자신만의 필요에 맞는 LLM Agent를 만들게 되길 바랍니다."

## 2026-04-09
- 타겟 유저는 비개발자와 프롬프트 엔지니어로 설정.
- 직관적인 'Teachable Machine' 컨셉의 GUI 기반 노드 에디터 채택.
- 생성물을 플랫폼 종속 없이 단독 실행 가능한 파이썬 파일(.py)로 Export 하는 기능을 핵심 기능으로 확정.
- 프론트엔드는 JS와 React, React Flow를 통하여 구현하기로 결정.
- 파이프라이닝 및 데이터 제어 핵심 로직은 블랙박스 형태의 라이브러리를 배제하고, 100% 이해 가능한 형태의 코드로 직접 구현하기로 결정.
- (2026-04-10) 이벤트 기반(Reactive) 아키텍처 채택: 폴링(While 루프) 방식이 아닌, 상태를 가지는 `Schema 오브젝트(Node/Edge)`와 상태 없이 비동기적으로 작업을 수행하고 파괴되는 `Task Consumer(Handler)` 그룹을 철저히 분리하여 설계.
  * 설계 배경: KENS(TCP/IP 네트워크 구현 프로젝트)의 책임 분리 설계 방식, 게임 개발의 Controller-View 페어 구조, 그리고 Kafka의 Topic-Consumer 아키텍처에서 얻은 인사이트를 바탕으로 디버깅과 확장이 용이하게 착안됨.
- (2026-04-10) 서비스 제공 형태(Business/Deploy Model) 논의:
  * [옵션 1] 중앙 집중형 웹 SaaS: 유저 접근성은 극대화되나, 테스트 실행 및 LLM 파이프라이닝을 전담하는 백엔드를 둬야 하므로 제작자의 지속적인 API 및 서버 비용(Cost) 이슈 발생 우려.
  * [옵션 2] 사용자 API 기반 독립형/오픈소스 앱: 사용자가 본인의 API 키를 입력하여 테스트하고 코드를 뱉어내는 완벽한 독립 형태. 비용이 없고 오픈소스 철학에 부합.
  * 현재 **옵션 2**를 지향하면서도 접근성을 잡을 수 있는 '100% 프론트엔드 기반의 서버리스(Serverless) 하이브리드 웹앱 형태'를 대안으로 적극 채택함.
  * **[중요 원칙]** 단, 클라이언트 사이드에서 모든 것을 처리하더라도 훗날 언제라도 프론트엔드-백엔드 구조로 분할(Migration) 재배포가 가능하도록, UI 컴포넌트 로직과 핵심 실행 엔진(Schema 상태 관리 및 Task Consumer)의 폴더 구조와 코드는 철저하게 물리적/논리적으로 분리하여 개발한다.
